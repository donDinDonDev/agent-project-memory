# v1.1 Gradle Java/Spring Evaluation Summary

Evaluation date: 2026-06-14

This public summary records the v1.1 validation pass for packaged CLI scans over pinned
Gradle Java/Spring targets and a selected Maven regression target. The v1.1 Gradle
expansion is additive over the stable v1.0 output marker, so this evaluation focused on
Gradle/Maven claim separation, deterministic output, resolving evidence references,
normalized paths, module and warning references, and absence of dynamic Gradle claims.

This was validation-only work. It did not change analyzer behavior, generated output
contracts, evidence semantics, fixtures, goldens, tests, release versioning,
publication state, or installation channels.

## Result

All selected targets generated deterministic output across repeated packaged CLI scans.
Structural validation passed for the evaluated outputs:

- all generated fact `evidence_ids` resolved to `evidence-index.jsonl` records;
- evidence IDs were unique within each generated output set;
- evidence paths were normalized repository-relative paths and resolved to local source
  files or documented path-only signals;
- module references resolved to emitted module IDs;
- warning reference lists resolved to emitted warning facts;
- Gradle-derived build facts stayed separate from Maven-derived build facts;
- Gradle build-file evidence used `build_file` records with bounded excerpts;
- generated outputs did not contain local absolute paths;
- guide wording did not claim Gradle execution, dependency resolution, plugin
  resolution, task graphs, custom `sourceSets`, `projectDir` remapping, included builds,
  Kotlin source analysis, or effective Gradle model reconstruction.

## Observations

- The v1.1 Gradle expansion behaved as an additive compatibility change:
  `project-map.json` remained on `schema_version: "1.0"`.
- Supported Gradle roots were represented as local build-file and standard-root
  observations rather than runtime Gradle model facts.
- Static settings include declarations produced module orientation and evidence when
  they stayed inside the documented literal subset.
- Unsupported or unsafe Gradle inputs degraded to warnings or scan diagnostics instead
  of creating unsupported source roots or dynamic build claims.
- The Maven regression target preserved the expected Maven-oriented module and evidence
  behavior while exercising the same packaged CLI artifact.

## Limitations

- This corpus is representative, not exhaustive.
- The evaluation did not run the selected projects' own Gradle or Maven builds or tests.
  It validated static packaged CLI scans only.
- No connector, network-service, SaaS, repository chat, generic RAG, LLM-core,
  generated-source scanning, package-manager distribution, release, tag, push, upload,
  or publication behavior was evaluated or added.
