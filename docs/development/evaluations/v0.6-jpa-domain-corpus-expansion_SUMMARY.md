# v0.6 JPA Domain Corpus Expansion Evaluation Summary

Evaluation date: 2026-06-09

## Scope

This public summary records the optional expanded corpus coverage for the published
v0.6 JPA/domain analyzer. The goal was to add real public Maven project coverage for
source-visible JPA signals that were not exercised by the original v0.6 real-project
corpus: `@Embeddable`, `@EmbeddedId`, `@IdClass`, `@Enumerated`, and `@Version`.

This evaluation did not change analyzer behavior, generated output schemas, evidence
semantics, fixtures, goldens, tests, release artifacts, tags, pushes, or published
release assets.

## Commands And Artifacts

The published local v0.6 packaged CLI already existed in the working checkout:

```sh
ls -l target/agent-project-memory-0.6.0.jar
```

Candidate checkouts were cloned into an external evaluation work directory and pinned
by recording `git rev-parse HEAD` in each checkout:

```sh
git clone --depth 1 https://github.com/BroadleafCommerce/BroadleafCommerce.git <external-evaluation-workdir>/broadleafcommerce
git clone --depth 1 https://github.com/vladmihalcea/high-performance-java-persistence.git <external-evaluation-workdir>/high-performance-java-persistence
git clone --depth 1 https://github.com/eclipse-ee4j/jakartaee-examples.git <external-evaluation-workdir>/jakartaee-examples
```

The packaged CLI scan commands were run twice for each selected target:

```sh
JAR=target/agent-project-memory-0.6.0.jar

java -jar "$JAR" scan <external-evaluation-workdir>/broadleafcommerce
java -jar "$JAR" scan <external-evaluation-workdir>/high-performance-java-persistence
java -jar "$JAR" scan <external-evaluation-workdir>/jakartaee-examples/tutorial/case-studies/dukes-forest/entities
java -jar "$JAR" scan <external-evaluation-workdir>/jakartaee-examples/tutorial/persistence/order
java -jar "$JAR" scan <external-evaluation-workdir>/jakartaee-examples/tutorial/jaxrs/rsvp
java -jar "$JAR" scan <external-evaluation-workdir>/jakartaee-examples/focused/persistence/optimisticLock
```

Generated-file hashes were captured outside the repository with `shasum -a 256` for:

```text
.project-memory/project-map.json
.project-memory/evidence-index.jsonl
.project-memory/endpoints.md
.project-memory/agent-guide.md
```

Each first-run hash file was compared to the matching second-run hash file with
`diff -u`.

A small local Python validator then traversed each `project-map.json` and
`evidence-index.jsonl` to check:

- generated output file presence;
- `schema_version: "0.6"`;
- missing `evidence_ids` references;
- duplicate evidence IDs;
- normalized repository-relative evidence paths;
- missing evidence source paths;
- normalized and existing `source_path` values;
- bad `module_id` and `target_module_id` references;
- missing `warning_ids` references;
- duplicate warning IDs.

No release artifact, checksum, tag, push, GitHub release, asset upload, dependency
update, Maven execution inside scanned targets, or analyzer implementation action was
performed.

## Observations

- The optional corpus expansion now exercises every target signal in generated v0.6
  output through pinned public Maven repositories.
- Embedded target linking stayed conservative. `@Embedded` and `@EmbeddedId` fields
  linked only to unique source-visible local `@Embeddable` facts; no runtime ORM
  resolution, provider metadata, database schema, or composite-key semantics were
  inferred.
- `@IdClass` output stayed bounded. The emitted signals preserved source-visible type
  literals and kept field matching and semantic reconstruction as `not_analyzed`.
- Relationship targets stayed conservative. Relationship entries preserved
  `target_resolution: "declared_type_only"` and `uncertainty:
  "target_type_not_resolved"` rather than claiming resolved entity targets.
- Missing annotation attributes remained absent or `null`; the generated output did not
  fill JPA provider defaults.
- The High-Performance Java Persistence target contains additional JPA examples under
  `src/test/java`, but the JPA/domain analyzer correctly limited entity facts to
  supported production source roots. Its production-source `@Enumerated` signal was
  useful; mapped-superclass non-identifier fields were not expanded into entity field
  facts by this v0.6 contract.

## Findings Triage

No release-blocking or follow-up-blocking finding was found. The stop conditions for
systemic v0.6 overclaiming, contract/evidence mismatch, evidence reference failures,
module reference failures, warning reference failures, required analyzer fix, schema
change, security-relevant implementation change, release action, or pinned checkout
instability were not triggered.

This expanded corpus remains optional and non-blocking. The published `v0.6.0` release
did not require this expanded corpus, and this follow-up did not discover a need for a
v0.6 implementation or contract correction.

## Residual Risks

- This evaluation improves real-project corpus coverage for v0.6 JPA/domain signals but
  is still not an exhaustive JPA corpus.
- Some selected targets are Maven JPA sample repositories rather than Spring Boot
  applications. They are useful for deterministic JPA annotation coverage but do not
  replace future Spring-heavy domain corpus expansion.
- Repository/entity inferred relations were not materially exercised by this follow-up;
  they were already covered by the original v0.6 evaluation and focused fixtures.
