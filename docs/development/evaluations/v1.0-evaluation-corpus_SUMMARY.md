# v1.0 Evaluation Corpus Summary

Evaluation date: 2026-06-12

This public summary records the v1.0 validation pass for packaged CLI scans over a
representative Java/Spring Maven corpus. The v1.0 schema marker is a compatibility
marker over the v0.9 output shape, so this evaluation focused on determinism, resolving
evidence references, normalized paths, module and warning references, document/code
separation, and practical guide usefulness.

This was validation-only work. It did not change analyzer behavior, generated output
contracts, evidence semantics, fixtures, goldens, tests, release versioning, publication
state, or installation channels.

## Result

All selected targets generated deterministic v1.0 output across repeated packaged CLI
scans. Structural validation passed for the evaluated outputs:

- all generated fact `evidence_ids` resolved to `evidence-index.jsonl` records;
- evidence IDs were unique within each generated output set;
- evidence paths were normalized repository-relative paths and resolved to local source
  files or path signals;
- module references resolved to emitted module IDs;
- warning and endpoint reference lists resolved to emitted warnings and endpoint facts;
- generated outputs did not contain local absolute paths;
- document chunks kept `content_status: "not_serialized"`;
- reconciliation rows stayed low-confidence uncertain inspection hints;
- `document` evidence stayed within the top-level `documents` object and did not become
  code, API, build, test, entity, quality, or runtime evidence;
- the configured local Markdown run recorded scan config as redacted scan metadata and
  did not create tool-config evidence records.

## Observations

- The v1.0 schema marker behaved as the intended compatibility marker: evaluated
  `project-map.json` files used `schema_version: "1.0"` while preserving the current
  output shape and evidence semantics.
- Source-visible Spring MVC endpoints, declared OpenAPI operations, generated/hidden
  warnings, JPA/domain facts, test facts, quality hints, local document facts, and scan
  metadata stayed in their documented owners.
- The OpenAPI/spec target exercised declared operation output without promoting spec
  operations into implemented endpoint facts.
- The JPA-heavy target exercised embeddables, entity facts, component volume, tests, and
  a large guide. The output remained structurally valid, though guide size should
  continue to be monitored on larger domain-heavy projects.
- The test-heavy target exercised many source-visible test facts and Spring test
  signals without introducing coverage, execution, CI, assertion, or runtime claims.
- The docs-heavy configured run provided useful volume for local Markdown headings,
  chunks, reconciliation hints, and redacted scan config metadata. The default
  local-document scope remains intentionally narrower.

## Findings Triage

No release-blocking finding was found in this evaluation.

Bounded follow-up candidates:

| Follow-up | Priority | Scope | Summary |
| --- | --- | --- | --- |
| Larger default-scope Markdown corpus | Low | Evaluation corpus only | Add future pinned projects whose default root `docs/**/*.md` or module README layout exercises dense local-document output without a custom include config. |
| Large-guide readability review | Low | Evaluation first, rendering only if justified | Continue monitoring `agent-guide.md` size on JPA-heavy and test-heavy projects before changing presentation caps or section density. |
| Broader OpenAPI corpus | Low | Evaluation corpus only | Add future pinned projects with multiple local specs or spec warning cases before making stronger OpenAPI corpus claims. |

## Limitations

- This corpus is representative, not exhaustive.
- The docs-heavy pass used a local Markdown include config to exercise configured
  document discovery volume; the default local-document scope remains narrower by
  design.
- The evaluation did not run the selected projects' own Maven builds or tests. It
  validated static packaged CLI scans only.
- No connector, network-service, SaaS, repository chat, generic RAG, LLM-core,
  generated-source scanning, Gradle, package-manager distribution, release, tag, push,
  upload, or publication behavior was evaluated or added.
