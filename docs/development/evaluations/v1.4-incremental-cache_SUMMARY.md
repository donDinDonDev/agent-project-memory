# v1.4 Incremental Cache Evaluation Summary

Evaluation date: 2026-06-15

This public summary records the v1.4 release-candidate validation pass for packaged CLI
full scans and opt-in incremental scans. The v1.4 incremental cache expansion is
additive over the stable v1.0 output marker, so this evaluation focused on full-scan
parity, cache miss and cache hit behavior, cache invalidation, selected profile
artifacts, cache metadata boundaries, deterministic output, and evidence separation.

This was validation-only work. It did not change analyzer behavior, generated output
contracts, evidence semantics, fixtures, goldens, tests, publication state, or
installation channels beyond the release-prep documentation and version alignment.

Selected validation targets:

| Target | Purpose |
| --- | --- |
| Maven regression fixture | Full scan, incremental warm-up, cache hit, and changed-input invalidation over a compact Maven Java/Spring fixture. |
| Gradle regression fixture | Full scan and incremental behavior over the bounded static Gradle Java/Spring layout. |
| Selected profile scan | Incremental behavior when `--agent-profile codex` participates in selected output fingerprints. |
| Repository self-check | Packaged CLI scan over this project to exercise larger local repository traversal and cache metadata boundaries. |

The generated project maps emitted `schema_version: "1.0"` and the expected base
`.project-memory/` artifacts:

```text
.project-memory/project-map.json
.project-memory/evidence-index.jsonl
.project-memory/endpoints.md
.project-memory/agent-guide.md
```


## Result

Release-candidate validation passed for the evaluated targets:

- full scan outputs were generated successfully;
- first incremental scans warmed the cache after full analysis;
- repeated incremental scans reused the existing generated output set for unchanged
  repository states;
- changed-input validation fell back to full analysis and refreshed cache metadata;
- selected profile output fingerprints covered `agent-profiles/manifest.json` and the
  selected profile Markdown file;
- no-profile scans kept unselected profile artifacts outside the selected output set;
- cache metadata contained only the documented metadata surface;
- generated outputs did not add cache status, output digest, or timing fields;
- generated facts continued to reference source-backed evidence records.

## Observations

- v1.4 incremental mode behaved as an additive compatibility change:
  `project-map.json` remained on `schema_version: "1.0"`.
- The cache layer stayed opt-in. Normal scans without `--incremental` did not depend on
  persistent cache state.
- Cache hit reuse kept whole-output-set parity for unchanged repository states.
- Cache miss, stale cache, and invalidation cases degraded to normal full analysis
  rather than creating partial or stale project facts.
- Cache files stayed execution metadata and did not create `evidence-index.jsonl`
  records.

## Findings Triage

No release-blocking finding remains open for this release-candidate evaluation.

## Limitations

- This validation is representative, not exhaustive.
- The evaluation did not run scanned projects' own Maven or Gradle builds.
- The evaluation did not add performance promises, timing data, remote cache behavior,
  daemon behavior, connector behavior, generated-source content scanning, release, tag,
  push, upload, or publication behavior.
